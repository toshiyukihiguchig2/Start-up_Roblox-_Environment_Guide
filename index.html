運

セットアップ後の運用

毎回の開発セッションは以下の手順で開始します。

  1. 1
    PowerShellを起動し、クローンしたリポジトリへ移動してVSCodeを開く
    powershell
    cd C:\Users\<ユーザー名>\source\repos\<project-name>
code .
  2. 2
    VSCodeのコンソール（PowerShell）でRojoを起動する
    powershell
    rojo serve
  3. 3
    Roblox Studioを起動し、制作中のバーチャル空間を選択する セットアップ済みのプロジェクトファイルを開いてください。
  4. 4
    Roblox StudioのプラグインからRojoを開き「Connect」を押下する VSCode側のコードがStudioに同期されます。
流

開発を進める流れ

  1. 1
    本番コードを書くsrc/ に本番コードを実装する
  2. 2
    テストを書くsrc/tests/ に対応するテストを書く
  3. 3
    動作確認Roblox Studio で Rojo 経由で同期して動作確認（1〜3を繰り返し）
  4. 4
    push & CI 自動テストGitHub に push → CI が自動テスト（結果をGitHubのActionsタブで確認）

4.0

GitHubの準備

1. GitHubアカウントの作成

  1. 1
    GitHub公式サイトにアクセスし、「Sign up」をクリックします。
  2. 2
    メールアドレス、パスワード、ユーザー名を入力して登録を完了させます。
  3. 3
    登録したメールアドレスに届く認証コードを入力して、アカウントを有効化してください。

2. Gitのインストールと初期設定

  1. 1
    Gitのインストール Git for Windows からインストーラーをダウンロードして実行します（設定はすべて「Next」でOK）。
  2. 2
    初期設定 PowerShellを開き、以下のコマンドで名前とメールアドレスを設定します。
    powershell
    git config --global user.name "Your Name"
git config --global user.email "your-email@example.com"

3. プロジェクト用リポジトリの作成

  1. 1
    GitHubのトップ画面左側、または右上の「＋」アイコンから 「New repository」 をクリックします。
  2. 2
    Repository name：プロジェクト名を入力（例：my-roblox-game）。
  3. 3
    Public / Private：チーム内のみなら「Private」、公開するなら「Public」を選択。
  4. 4
    Initialize this repository with：
    • Add a README file：チェックを入れる
    • Add .gitignore：「Roblox」で検索して選択
    • Choose a license：「MIT License」などを選択
  5. 5
    「Create repository」 をクリックして完了。
  6. 6
    作成したリポジトリの緑色の 「<> Code」 ボタンをクリックし、「HTTPS」タブのURL（https://github.com/<ユーザー名>/<project-name>.git）をコピーします。このURLを次のクローン手順で使用します。

4. 認証の設定（重要）

⚠️
最近のGitHubはパスワードだけで git push ができません。以下のいずれかの設定が必要です。
方法説明難易度
GitHub Desktopツール側でログインするだけで済む。初心者に最もおすすめ⭐ 簡単
Personal Access Token (PAT)ブラウザの「Settings」→「Developer settings」から発行し、パスワードの代わりに使用⭐⭐ 普通

4.2

Rojoのインストール

実行環境：PowerShell

インストール方法は2つあります。Windowsはバイナリダウンロードが安定しておすすめです。

方法A：cargo でインストール（Rust 環境が必要）
powershell
cargo install rojo

方法B：バイナリをダウンロード（おすすめ）

  1. 1
    Rojo 公式リリースページを開き、最新版の Assets から Windows用 zip（例：rojo-x.x.x-windows-x86_64.zip）をダウンロードして解凍する。
  2. 2
    中にある rojo.exe を任意のフォルダ（例：C:\Users\<ユーザー名>\AppData\Local\Programs\Rojo）に置く（フォルダは自分で作成して OK）。
  3. 3
    そのフォルダをPATHに追加する
    • Windowsキー →「環境変数」→「環境変数の編集」
    • 「Path」を選択 →「編集」
    • C:\Users\<ユーザー名>\AppData\Local\Programs\Rojo を追加
  4. 4
    PowerShellを新しいウィンドウで開いて動作確認する
    powershell
    rojo --version   # バージョンが表示されればOK

4.4

Roblox StudioでRojoプラグインに接続

Rojoプラグインのインストール

  1. 1
    Roblox Studio を開く。
  2. 2
    上部メニューの Plugins（プラグイン） をクリック。
  3. 3
    Manage Plugins（プラグイン管理） を開く。
  4. 4
    右上の「Find Plugins（プラグインを探す）」または「＋」をクリック。
  5. 5
    ツールボックスの検索欄に 「Rojo」 と入力し、公式のRojoを選んで「Install（インストール）」。
  6. 6
    インストール後、Studio中央の Plugins（プラグイン） タブに Rojo ボタンが表示される。

Rojoプラグインの動作確認と接続

VSCode側で rojo serve が動いている状態で、Studio の Rojo ボタンを押すと：

  1. 1
    Rojo の窓が開き、ポート番号 34872 が表示される（自動）。
  2. 2
    「Connect」 をクリック → VSCode の src/ が Studio の ReplicatedStorage/Source に同期される。
⚠️
Roblox Studio の Rojo プラグインと VSCode で参照している Rojo CI のバージョンが一致しないと接続できないことがあります。その場合はどちらかのバージョンをアップグレードまたはダウングレードしてください。

4.5

TestEZの構築

Wally を使って TestEZ を導入します。

Wallyのインストール（ブラウザで直接ダウンロード・おすすめ）

  1. 1
    Wallyをダウンロード Wally Releases を開き、Assets にある wally-windows.exe を含む zip をダウンロードして解凍する。
  2. 2
    wally.exe を配置 wally.exe を任意のフォルダ（例：C:\Users\<ユーザー名>\AppData\Local\Programs\Wally）に置く（フォルダは自分で作成して OK）。
  3. 3
    PATHへ追加 Windowsキー →「環境変数」→「環境変数の編集」→「Path」を選択 →「編集」→ C:\Users\<ユーザー名>\AppData\Local\Programs\Wally を追加。
  4. 4
    動作確認
    powershell
    wally --version

wally.toml の作成

実行環境：VSCode

  1. 1
    VSCode の左側でプロジェクトルートを右クリックし「新しいファイル」を選択。
  2. 2
    ファイル名 wally.toml を入力して作成する。
  3. 3
    以下の内容を貼り付ける（TestEZ 導入用）
    toml
    [package]
name = "<GitHubユーザー名>/<project-name-lowercase>"
version = "0.1.0"
registry = "https://github.com/UpliftGames/wally-index"
realm = "shared"

[dependencies]
TestEZ = "roblox/testez@0.4.1"
    💡
    name は GitHub のユーザー名に合わせて変更してOK。
    プロジェクト名は 小文字で設定してください（例：My-Roblox-Game → my-roblox-game）。
  4. 4
    Wally を適用する
    powershell
    wally install   # プロジェクト内に Packages/ フォルダが生成されればOK

TestEZの実行準備

まず default.project.json に Packages のマッピングを追加します。

json
{
    "name": "<project-name>",
    "tree": {
        "$className": "DataModel",
        "ReplicatedStorage": {
            "$className": "ReplicatedStorage",
            "shared": {
                "$path": "src/shared"
            },
            "Packages": {
                "$path": "Packages"
            }
        },
        "ServerScriptService": {
            "$className": "ServerScriptService",
            "Server": {
                "$path": "src/server"
            },
            "TestRunner": {
                "$path": "src/runner"
            }
        }
    }
}

次にテストコードを作成して動作確認します。

  1. 1
    src/tests/ フォルダを作成する。
  2. 2
    src/tests/example.spec.lua を作成し、サンプルコードを記述する
    lua
    return function()
    describe("math", function()
        it("adds numbers", function()
            expect(1 + 1).to.equal(2)
        end)
    end)
end
    保存すると、Studio側の ReplicatedStorage/Source/tests に自動同期されます。
  3. 3
    Roblox Studio で TestEZ を実行する Rojo 接続中の Studio 上で、TestEZ の UI がない場合は Command Bar で以下を実行：
    lua
    local TestEZ = require(game.ReplicatedStorage.Packages.TestEZ)
local results = TestEZ.TestBootstrap:run({game.ReplicatedStorage.Source.tests})
print(results)

TestRunner（UI）の作成

「プレイ（F5）」を押すだけでテストが走るよう、TestRunner を設定します。

  1. ①
    src/runner/ フォルダを作成する
    src/ ├ tests/ └ runner/ ← 新規作成
  2. ②
    src/runner/TestRunner.server.lua を作成
    lua
    local TestEZ = require(game.ReplicatedStorage.Packages.TestEZ)
local results = TestEZ.TestBootstrap:run({
    game.ReplicatedStorage.Source.tests
}, TestEZ.Reporters.TextReporter)
print("===== TestEZ Finished =====")
  3. ③
    Rojo の default.project.json に runner を追加する
    json
    {
    "name": "<project-name>",
    "tree": {
        "$className": "DataModel",
        "ReplicatedStorage": {
            "$className": "ReplicatedStorage",
            "Source": {
                "$path": "src"
            },
            "Packages": {
                "$path": "Packages"
            }
        },
        "ServerScriptService": {
            "$className": "ServerScriptService",
            "TestRunner": {
                "$path": "src/runner"
            }
        }
    }
}
  4. ④
    Studio で TestEZ 実行を確認 「プレイ（F5）」を押すと「出力」にテスト実行結果が表示されます。
4.6

GitHub Actions の自動テスト（CI）を設定

GitHub Actions は GitHub のサーバー上で TestEZ を実行します。必要な構成は以下の3つです。

🔧
Foreman
Roblox CLI のパッケージ管理ツール
📦
Wally
依存パッケージ管理（TestEZ等）
⚙️
workflow
push のたびに自動テストを実行
ℹ️
前提：VSCode に GitHub Actions プラグインをインストール済みであること。

foreman.toml の追加

実行環境：VSCode

プロジェクトルートに foreman.toml を作成し、以下を記載します。

toml
[tools]
wally = { source = "UpliftGames/wally", version = "0.3.2" }
rojo  = { source = "rojo-rbx/rojo",       version = "7.6.1"  }
💡
version はインストール済みの実際のバージョンを設定してください。不明な場合はターミナルで確認できます。
wally --version  /  rojo --version

test.project.json の追加

プロジェクトルートに test.project.json を作成します。これは GitHub Actions が TestEZ を実行するための「テスト専用の Rojo プロジェクト」です。

json
{
    "name": "tests",
    "tree": {
        "$path": "src/tests"
    }
}

GitHub Actions の workflow を追加

実行環境：VSCode

.github/workflows/test.yml を作成し、以下を記載します。

yaml
name: Test
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Foreman
        run: |
          curl -L https://github.com/Roblox/foreman/releases/latest/download/foreman-linux-x86_64.zip -o foreman.zip
          unzip foreman.zip
          chmod +x foreman
          sudo mv foreman /usr/local/bin/

      - name: Install tools (Wally & Rojo)
        run: foreman install

      - name: Add Foreman tools to PATH
        run: echo "$HOME/.foreman/bin" >> $GITHUB_PATH

      - name: Install Wally packages
        run: wally install

      - name: Build place
        run: rojo build test.project.json -o test.rbxl
✅
GitHub に push すると、Actions タブに「Test」ジョブが走り、成功・失敗が GitHub 上で確認できます。
除

.gitignore について

⚠️
以下のファイルは 絶対にGitに含めないでください。
除外対象理由
.rbxl / .rbxmRobloxバイナリファイル。差分が取れず容量も巨大になるため絶対に除外。Git に入れるとリポジトリが壊れることがある
out/ / build/Rojo の出力・生成物。コミット不要
.vscode/個人の環境依存設定。チーム開発で競合の原因になる
Packages/Wallyで管理する依存ファイル。wally install で再構築できるためコミット不要
coverage/ などCI の一時ファイル。生成物はコミットしない
注

注意点

  • .rbxl や .rbxm などのバイナリファイルは Git に含めない
  • src/ 以下のコードが開発の中心。Studio 側でスクリプトを直接編集しない
  • Rojo のマッピング設定（default.project.json）を変更した場合はチームへの共有が必須
  • GitHub Actions の設定ファイル（.github/workflows/）は削除しない
  • テストコードは src/tests/ に配置する
  • Rojo プラグインと CI のバージョンは揃えること
